# Policy — Contributor Calendar Usage and Hour Tracking
Status: Draft v1.0
Owner: Ed (Steward)
Applies To: Contributors, Creators, Project Operators, OpenClaw support workflows
Last Updated: 2026-02-23
## Purpose
Provide a lightweight, fair way to use calendar events as part of work and hour tracking for contributors (including creators like Nick), without creating excessive admin burden.
This policy supports:
\- visibility into work commitments
\- planning and coordination
\- better budgeting and staffing
\- documented activity for project operations
It does **not** assume every calendar event equals billable/compensable work by default.
## Core Principles
1. **Calendar supports accountability; it does not replace trust**
2. **Track commitments and work sessions consistently**
3. **Use clear categories for work types**
4. **Minimize manual overhead**
5. **Respect creator workflows while preserving operational visibility**
## Policy Statement
Contributors performing scheduled work for projects should use shared calendar events (or approved equivalents) to represent planned and completed work blocks when hour tracking is required for operational, budget, or compensation purposes.
The calendar is a planning and evidence layer. Final compensation decisions may also consider deliverables, outcomes, and approved agreements.
## Who This Applies To
\- creators with scheduled production sessions
\- contributors with assigned work blocks
\- project managers / operators coordinating team activity
\- stewards reviewing capacity and budget alignment
## What Should Be Tracked on Calendar
### Track (when relevant to project ops)
\- production sessions
\- editing blocks
\- writing / scripting sessions
\- recording sessions
\- review meetings
\- client meetings
\- planning sessions
\- office hours / collaboration blocks
\- delivery deadlines (milestone markers)
### Do Not Require Tracking (unless explicitly needed)
\- incidental thinking time
\- personal/admin tasks unrelated to project
\- unscheduled ad hoc work under threshold (team can define threshold)
\- private prep that creator chooses not to disclose in detail
## Event Categories (Recommended Tags)
Use simple tags in event title or description:
\- `\[MEET\]` Meetings
\- `\[WORK\]` Focused production/work block
\- `\[REVIEW\]` Review / approvals
\- `\[CLIENT\]` Client-facing
\- `\[ADMIN\]` Ops/admin
\- `\[DELIVERABLE\]` Milestone / deadline marker
Example:
`\[WORK\] TLN episode outline \+ script structuring (2h)`
## Minimum Event Data for Trackable Work
Trackable events should include:
\- project / initiative name
\- work type (meeting/work/review/etc.)
\- duration
\- owner/contributor
\- short description of intended output
\- status if relevant (planned / completed / rescheduled)
Optional but useful:
\- linked task/doc
\- episode/project code
\- collaborators involved
## Planned vs Completed Work
Calendar events should reflect reality.
If plans change:
\- reschedule or mark canceled
\- adjust duration if materially different
\- avoid leaving inaccurate “ghost work” blocks
If using calendar data for reporting, teams should distinguish:
\- **Planned Hours**
\- **Completed Hours**
\- **Meeting Hours**
\- **Production Hours**
## Creator-Friendly Tracking Standard
For creators (e.g., Nick), the goal is not surveillance.
The goal is operational clarity.
Recommended approach:
\- track major production blocks and meetings
\- tie blocks to outputs (episode, asset, review, etc.)
\- keep descriptions short and useful
\- let detailed creative process stay in creator workflow tools if preferred
## Role Responsibilities
### Contributor / Creator
\- maintain trackable events when required by project policy
\- keep event timing reasonably accurate
\- link to outputs/tasks when possible
### Project Lead / Operator
\- define what categories are required
\- review hours/capacity against project needs
\- avoid overburdening contributors with unnecessary tracking
### Ops / Steward
\- set organization-wide standards
\- align tracking with budget, compensation, and governance requirements
\- audit for consistency, not micromanagement
### OpenClaw (Assistive)
Can:
\- draft event descriptions from task lists
\- generate weekly summaries from calendar \+ task data
\- flag missing metadata
\- produce capacity snapshots
Cannot:
\- infer completed work without evidence
\- assign compensable hours unilaterally
\- replace project lead approval where needed
## Hours Tracking and Compensation
Calendar entries may be used as one input for:
\- operational reporting
\- budgeting
\- staffing/capacity planning
\- contributor review conversations
If compensation depends on hours:
\- define approval workflow explicitly
\- define rate / point logic separately
\- identify exceptions (flat fee, milestone-based, RevPoint-based, etc.)
Do not mix compensation policy into calendar policy unless documented.
## Privacy and Sensitivity
\- Use shared project calendars for project work visibility
\- Avoid exposing private personal schedule details
\- For sensitive projects, use generalized event descriptions if needed
## Quality Standards
Good event:
\- clear project
\- clear type
\- realistic duration
\- useful outcome context
Bad event:
\- “Work”
\- no project
\- no duration clarity
\- never updated after changes
## Suggested Weekly Operating Rhythm
\- Contributor updates upcoming week blocks
\- Project lead reviews capacity and conflicts
\- OpenClaw (optional) generates summary for ops review
\- End-of-week compare planned vs completed for learning (not punishment)
## Exceptions
Some roles may be output-based rather than hour-tracked.
If so, define:
\- deliverable milestones
\- review cadence
\- minimum scheduling visibility expectations
## Review Cadence
Review after:
\- first month of use
\- first compensation cycle using calendar data
\- onboarding of first creator cohort using this policy
