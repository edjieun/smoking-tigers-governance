## Context

DEC-20260324-003 designated Notion as the canonical source for projects, tasks, and meeting records. In practice, Notion's project management layer (the Exec Team Task Tracker and STE Initiatives DB) lacks the workflow state management, agent-writeable issue tracking, and cross-project visibility needed to support the current operational tempo.

On 2026-04-17, Linear was deployed as a dedicated issue tracking and project coordination platform for Smoking Tigers. The following was established in Linear:

- **Team:** Smoking Tigers (key: `STE`)
- **Projects:** Trade Like Nick (TLN), AI Agent Infrastructure, Operations, ST:AI Landing Page, plus additional projects for active workstreams
- **Initiatives:** 15 initiatives migrated from the Notion STE Initiatives DB, representing Active/Activating/Prototyping lifecycle items
- **Issues:** Seeded from OpenClaw workspace memory, TASKS.md, and Notion ExecDB data
- **Labels:** `agent-task`, `decision`, `tln`, `infrastructure`, `blocked`
- **API access:** Linear GraphQL API authenticated via personal API key stored in workspace credentials

Linear is now the primary place where issues are created, assigned, tracked to completion, and closed — including issues generated by AI agents.

## Decision

### 1. Linear Adopted as Canonical Issue and Project Coordination Layer

Linear is adopted as the canonical source for:
- **Issue tracking** — individual tasks, action items, and work units
- **Project coordination** — grouping issues into projects and linking projects to initiatives
- **Agent task visibility** — issues created by AI agents (labelled `agent-task`) to provide an audit trail of agent activity
- **Workflow state management** — Backlog → Todo → In Progress → In Review → Done / Cancelled

Notion remains canonical for:
- **Meeting records** — agendas, notes, decisions captured in meeting DBs
- **High-level project strategy** — the STE Initiatives DB as a strategic registry
- **Structured org records** — contributor directory, RevPoint ledger, calendar tracking

### 2. Responsibility Split: Linear vs Notion

| Content Type | Canonical Source | Notes |
|---|---|---|
| Individual tasks and action items | **Linear** | Replaces Notion Exec Team Task Tracker for new items |
| Project-level coordination | **Linear** | Projects, cycles, milestones |
| Initiatives (strategic layer) | **Linear** (primary) + Notion STE Initiatives DB (archive) | Linear is the live view; Notion is the original record |
| Meeting records | **Notion** | Agendas, notes, follow-up items → Linear issues |
| Agent-generated tasks | **Linear** (label: `agent-task`) | Agents create issues, not Notion records |
| Contributor records | **Notion** | Directory, RevPoints, agreements |
| Financial records | **Notion** | RevPoint ledger, expense tracking |

**Migration note:** Existing open tasks in the Notion Exec Team Task Tracker are not required to be migrated to Linear immediately. New tasks go to Linear. Notion task tracker becomes read-only legacy for historical reference.

### 4. Agent Rules for Linear

**L-1: Agent-Created Issues Must Be Labelled**
All issues created by AI agents must carry the `agent-task` label. This provides a human-readable audit trail of agent work.

**L-2: Agents Close What They Open**
When an agent creates a Linear issue to track a task, that agent (or a designated follow-on agent) is responsible for closing or updating the issue when the task completes or is blocked. Agents do not create issues and abandon them.

**L-3: No Duplicate Issues**
Before creating a new issue, agents must check whether an equivalent issue already exists in the relevant project. Use `LINEAR_LIST_LINEAR_ISSUES` or a search to verify. Duplicate issues are cancelled, not deleted.

**L-4: Issues Link to Source**
When creating an issue from an external source (Notion task, Discord message, decision), the issue description must include a reference to the source. This preserves traceability.

**L-5: Priority Must Be Set**
All issues created by agents must have a priority set. Agents must not leave priority as `No priority`.

**L-6: Morning Briefing Integration**
The daily morning briefing cron (7:00 AM Pacific) must include open and blocked Linear issues from the prior day alongside Notion and Discord activity. Issue: STE-16 tracks this implementation.

### 6. Updated Fragmentation Rules

The following rules from DEC-20260324-003 are updated:

**F-1 (amended): Single Source of Truth per Content Type**
- Governance decisions → GitHub repo only
- **Issue tracking and task coordination → Linear only**
- Meeting records → Notion only
- Strategic initiative registry → Notion (STE Initiatives DB) + Linear (live view)
- Agent configuration → OpenClaw workspace only
- Production assets and reports → Google Drive only

**F-5 (new): Linear as Agent Task Surface**
- AI agents create tasks in Linear, not Notion
- Notion task tracker is read-only legacy after 2026-04-17
- Agent-generated issues carry the `agent-task` label

### 7. Updated Agent Retrieval Order

1. `memory_search` (OpenClaw workspace + daily files)
2. `qmd query` — hybrid search across governance repo, project files, vault
3. **Linear GraphQL API** — issue and project status (when task context is needed)
4. Notion API — meetings, contributors, strategic context
5. GitHub governance repo via `read` tool (specific doc)
6. Google Drive API (specific document fetch)
7. If not found → flag as knowledge gap to Scout / Ed

---

## Rationale

- Notion's task tracker is human-centric and does not support AI agent write workflows well — no standardized states, no label taxonomy, no cycle/milestone tracking
- Linear provides a structured, API-first workflow with consistent states, labels, and priority that agents can reliably read and write
- Separating issue tracking (Linear) from structured records (Notion) follows the fragmentation prevention principle: one source of truth per content type
- The `agent-task` label creates a lightweight audit trail that surfaces agent activity in the same view as human-created work — no separate logging layer required
